## SOP & CORS
오늘은 브라우저의 보안 정책인 <span style="color: red; font-weight: bold;">SOP(Same-Origin Policy)</span>와 <span style="color: red; font-weight: bold;">CORS(Cross-Origin Resource Sharing)</span>에 대해 알아보자.
매번 블로그나 유튜브를 찾아보며 이해를 하고 나중에 보면 새로운 느낌은 뭘까...

<br>

### SOP
먼저 SOP(Same-Origin Policy)부터 살펴보자. 
SOP는 초기에 자바스크립트가 다른 출처의 DOM에 접근하는 것을 차단하기 위해 도입된 정책이다. 이후 <u>브라우저의 쿠키, 스토리지, AJAX 요청 등 다양한 민감 리소스 접근을 제어</u>하는 핵심 보안 모델로 확장되었다.

#### Origin 
먼저 브라우저가 동일 출처(Origin)에 대해 어떻게 판단하는지부터 알아보자. <br>
브라우저는 **프로토콜, 도메인, 포트** 이렇게 3가지 요소가 모두 일치하는지를 기준으로 동일 출처 여부를 판단한다.

예시로 https://k-co-co.github.io/ 도메인으로 예시를 들어보자. <br>
여기서 프로토콜은 **https**, 도메인은 **k-co-co.github.io**, 포트는 **443**이다. <br>
예시로 https://k-co-co.github.io:<span style="color: red; font-weight: bold">8443</span> 도메인은 포트(443 ≠ 8443)가 다르기 때문에 동일한 출처가 아니게 된다.

<br>

출처가 다르더라도 SOP에 의해 차단되지 않고 허용되는 경우가 몇 가지 존재한다.
* \<script\> 태그를 통한 다른 출처의 JS 삽입
* \<link rel="stylesheet"\> 태그를 통한 다른 출처의 CSS 삽입 (올바른 Content-Type 헤더 설정 필요)
* \<img\> 태그를 통한 다른 출처의 이미지 삽입 (\<video\>, \<audio\> 태그도 마찬가지)
* \<iframe> 태그를 통한 다른 출처의 DOM 삽입 (단, JavaScript로 다른 출처 iframe 내부 DOM에는 접근 불가)
* \<form> 태그를 통한 요청 전송

<br> 

여기까지 SOP에 대한 설명이다. <br>
<u>SOP는 출처(프로토콜, 도메인, 포트)가 다른 리소스에 대한 접근을 제한하는 브라우저의 핵심 보안 정책</u>이다.
그러나 현대 웹 서비스에서는 API서버, CDN, 외부 서비스 등 다른 출처와의 통신이 필수적이다. 이에 따라 <u>서버가 명시적으로 교차 출처 요청을 허용할 수 있도록 만든 메커니즘이 CORS</u>다.

<br><br><br>

### CORS
앞에서 언급했듯이 CORS(Cross-Origin Resource Sharing)는 SOP에 의해 제한되는 교차 출처 요청을 서버의 명시적인 허용에 따라 가능하게 하는 매커니즘이다.



#### CORS 동작 원리
CORS의 동작 원리에 대해 알아보자. 단순 요청과 Preflight 요청, 두 가지로 나눠볼 수 있다.

##### 1️⃣ Simple Request
먼저 브라우저는 교차 출처 요청을 보낼 때 Origin 헤더를 추가하여 전송한다.
```
GET / HTTP/1.1
Origin: https://k-co-co.github.io/
```
그러면 서버는 다음과 같이 응답을 뱉는다.
```
HTTP/1.1 200 OK
  .. 중략 ..
Access-Control-Allow-Origin: https://k-co-co.github.io/
```
이렇게 응답을 받으면 브라우저는 해당 응답을 스크립트에서 접근할 수 있도록 허용한다. <br>
반대로 Access-Control-Allow-Origin 헤더가 없거나 값이 일치하지 않을 경우 응답 데이터에 대한 스크립트 접근을 제한한다.


💡 CORS에서 교차 출처 요청의 허용 여부를 판단하고 차단하는 주체는 서버가 아니라 <span style="color: red; font-weight: bold">브라우저</span>이다.


##### 2️⃣ Preflight Request
다음은 조금 더 복잡한 Preflight Request이다. <br>
이는 브라우저가 실제 요청에 앞서, <u>해당 메서드와 헤더가 허용되는지 서버에 미리 확인하는 과정</u>을 거친다. <br>

Preflight Request는 GET, HEAD를 제외한 메서드를 사용하거나 POST 요청이라도 Content-Type이 application/json과 같이 단순 타입이 아닌 경우 발생한다. <br>
다만 모든 요청마다 Preflight가 수행되는 것은 아니다. 브라우저는 서버 응답에 포함된 Access-Control-Max-Age 값을 기준으로 결과를 일정 시간 캐싱한 후 동일한 조건의 요청에 대해서는 Preflight를 생략한다.

먼저 브라우저는 서버에 OPTIONS 요청을 아래와 같이 보낸다.
```
OPTIONS /api/user HTTP/1.1
Origin: https://k-co-co.github.io
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type
```
그러면 서버는 아래와 같이 응답을 뱉는다.
```
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://k-co-co.github.io
Access-Control-Allow-Methods: POST, GET
Access-Control-Allow-Headers: Content-Type
Access-Control-Max-Age: 3600
```
허용할 메서드는 GET/POST, 허용 헤더는 Content-Type, Preflight 캐싱은 3600초 
이후 브라우저는 실제 요청을 전송하고 데이터를 주고받으며, 단순 요청과 마찬가지로 스크립트로 해당 응답에 접근할 수 있도록 허용한다.


##### ⚡️ Access-Control-Allow-Credentials
CORS에서 유의깊게 봐야할 헤더가 있다. 바로 <span style="color: red; font-weight: bold">Access-Control-Allow-Credentials</span>이다. 
<u>해당 헤더는 교차 출처 요청 시 브라우저가 쿠키나 인증 정보를 요청에 포함할 수 있는지를 결정</u>한다.

아래는 fetch API를 수행하는 예시이다. <br>
요청 시 <u>credentials: 'include'</u>를 넣으면 쿠키나 인증정보를 요청에 포함하도록 지시한다.
```
fetch('https://api.example.com/data', {
  method: 'GET',
  credentials: 'include' // 쿠키/인증 정보를 포함
})
```
만약 쿠키/인증정보를 포함을 허용한 서버의 경우 아래와 같이 응답을 준다.
```
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://k-co-co.github.io
Access-Control-Allow-Credentials: true
Content-Type: application/json
...(생략)...
```
요청이 Preflight를 필요로 하는 경우, 브라우저는 먼저 Preflight를 수행하고, 서버가 허용하면 실제 요청에 쿠키/인증 정보를 포함하여 전송한다. 단순 요청인 경우에는 바로 쿠키/인증 정보를 포함한 요청을 보낸다.

<br>

💫 Access-Control-Allow-Origin이 *이 아닌 <u>특정 도메인으로 설정</u>되어야한다.
